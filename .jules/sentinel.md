## 2024-05-24 - Rate Limiting & Input Sanitization
**Vulnerability:** Chat input and API keys lacked max-length limitations, exposing the application to potential client-side DoS or buffer overflow via arbitrarily large strings. In addition, the `fetch` API request to Gemini was vulnerable to HTTP parameter injection (API key wasn't URL-encoded) and lacked an execution timeout, which could cause application hangs on unresponsive connections. Furthermore, the chat form lacked a debounce or state-lock, enabling users to spam requests and unnecessarily drain API limits.
**Learning:** Even seemingly closed ecosystem components like client-side forms and API communication routines must be fortified with defensive boundaries (length limits, proper encoding, and fail-safes) to maintain reliability and security, particularly when calling metered third-party endpoints.
**Prevention:** Always use `maxLength` on user input fields, implement robust `try/finally` blocks with boolean loading state locks to block concurrent request spam, strictly `encodeURIComponent` injected URL strings, and wrap `fetch` calls with `AbortController` timeouts.
